# Crawler Service Design

## Summary

The crawler service is the intake layer of the pipeline — it's what actually looks at the filesystem. The network share where healthcare data arrives uses directory naming as implicit metadata: the path structure encodes which project a delivery belongs to, which workplan, which data provider, which version, and whether QA has been signed off. The crawler reads those paths, extracts that metadata, takes stock of what SAS files are present, and registers each delivery with the registry API. It does not read file contents — it's a scanner, not a processor.

The approach is a strict split between pure logic and side effects. Parsing, fingerprinting, and manifest construction are all stateless functions with zero I/O — easy to test in isolation, easy to reason about. The filesystem walking, HTTP posting, and file writing are isolated in a thin imperative shell. Every crawl produces a manifest file on disk capturing a point-in-time snapshot of what was found; these exist as audit evidence independent of the registry's own state. The crawler is designed to run repeatedly and safely: re-scanning the same unchanged delivery produces identical output with no registry side effects.

## Definition of Done

1. **A path parser** (`crawler/parser.py`) that extracts structured metadata (request_id, project, request_type, workplan_id, dp_id, version) from directory paths, with configurable dp_id exclusions and isolated for unit testing.

2. **A filesystem crawler** (`crawler/main.py`) that walks configured scan_roots, detects QA status from directory naming (msoc vs msoc_new), counts files/sizes, computes fingerprints, and POSTs findings to the registry API.

3. **Crawl manifests** written to `pipeline/crawl_manifests/<delivery_id>.json` on every scan, capturing point-in-time evidence of what was seen (file inventory, parsed metadata, fingerprint).

4. **Error manifests** written for unparseable paths, providing audit trail for parse failures.

5. **Retry + abort**: exponential backoff on HTTP failures (3 attempts, 2s/4s/8s), abort entire run if API unreachable.

6. **Structured JSON logging** as the project's first logging pattern.

7. **Idempotent**: safe to re-run at any interval without side effects.

8. **Tests**: comprehensive unit tests for parser, integration tests for crawler logic, following existing project conventions.

**Out of scope:** converter, registry API changes, SAS file reading, cron setup.

## Acceptance Criteria

### crawler.AC1: Path Parser
- **crawler.AC1.1 Success:** Standard path `/requests/qa/mkscnr/packages/soc_qar_wp001/soc_qar_wp001_mkscnr_v01/msoc` returns correct metadata (request_id=`soc_qar_wp001`, project=`soc`, request_type=`qar`, workplan_id=`wp001`, dp_id=`mkscnr`, version=`v01`, qa_status=`passed`)
- **crawler.AC1.2 Success:** Path with `msoc_new` terminal directory returns qa_status=`pending`
- **crawler.AC1.3 Success:** dp_id at boundary lengths (3 and 8 characters) parses correctly
- **crawler.AC1.4 Success:** Version variants (`v01`, `v1`, `v10`) parse correctly
- **crawler.AC1.5 Success:** Paths from different scan_roots parse correctly with correct scan_root in result
- **crawler.AC1.6 Failure:** dp_id shorter than 3 or longer than 8 characters returns parse error
- **crawler.AC1.7 Failure:** Missing version segment returns parse error with descriptive reason
- **crawler.AC1.8 Failure:** Path not ending in `msoc` or `msoc_new` returns parse error
- **crawler.AC1.9 Edge:** Excluded dp_id (e.g., `nsdp`) returns `None` (not an error)
- **crawler.AC1.10 Edge:** Request ID with more than 3 underscore-separated segments parses correctly (everything before dp_id+version match is request_id)

### crawler.AC2: Filesystem Crawler
- **crawler.AC2.1 Success:** Crawler discovers all `msoc` and `msoc_new` directories under configured scan_roots
- **crawler.AC2.2 Success:** File inventory includes all `.sas7bdat` files with correct size_bytes and modified_at
- **crawler.AC2.3 Success:** Crawler POSTs valid DeliveryCreate payload to registry API for each parsed delivery
- **crawler.AC2.4 Success:** Crawler processes multiple scan_roots in a single run
- **crawler.AC2.5 Failure:** Non-existent scan_root is logged and skipped (does not abort entire crawl)
- **crawler.AC2.6 Edge:** Empty delivery directory (no .sas7bdat files) is still processed with file_count=0

### crawler.AC3: Crawl Manifests
- **crawler.AC3.1 Success:** Manifest written to `pipeline/crawl_manifests/<delivery_id>.json` with all required fields
- **crawler.AC3.2 Success:** Manifest `delivery_id` matches SHA-256 hex of `source_path`
- **crawler.AC3.3 Success:** Manifest `files` array contains complete inventory with filename, size_bytes, modified_at
- **crawler.AC3.4 Success:** Re-crawling same unchanged delivery overwrites manifest with identical content (idempotent)
- **crawler.AC3.5 Success:** Manifest includes `crawler_version` and `crawled_at` timestamp

### crawler.AC4: Error Manifests
- **crawler.AC4.1 Success:** Unparseable path produces error manifest in `pipeline/crawl_manifests/errors/`
- **crawler.AC4.2 Success:** Error manifest contains raw_path, scan_root, error reason, and crawler_version
- **crawler.AC4.3 Success:** Error manifest filename is deterministic hash of raw_path (idempotent on re-crawl)
- **crawler.AC4.4 Edge:** Excluded dp_ids do NOT produce error manifests (they are expected, not errors)

### crawler.AC5: Retry and Abort
- **crawler.AC5.1 Success:** Successful POST on first attempt returns response and continues
- **crawler.AC5.2 Success:** Transient 5xx triggers retry with backoff (2s, 4s, 8s), succeeds on later attempt
- **crawler.AC5.3 Failure:** All 3 retries exhausted raises RegistryUnreachableError
- **crawler.AC5.4 Failure:** Crawler exits non-zero when RegistryUnreachableError is raised
- **crawler.AC5.5 Edge:** 4xx response is NOT retried (immediate failure)

### crawler.AC6: Structured JSON Logging
- **crawler.AC6.1 Success:** Log output is valid JSON lines (one JSON object per line)
- **crawler.AC6.2 Success:** Each log entry includes timestamp, level, and message fields
- **crawler.AC6.3 Success:** Contextual fields (scan_root, source_path, delivery_id) present when available
- **crawler.AC6.4 Success:** Logs written to both file (log_dir) and stderr

### crawler.AC7: Idempotency
- **crawler.AC7.1 Success:** Running crawler twice on same directory tree produces identical registry state and manifests
- **crawler.AC7.2 Success:** Unchanged fingerprint means registry `last_updated_at` is not modified on re-crawl

### crawler.AC8: Tests
- **crawler.AC8.1 Success:** Parser unit tests cover all AC1 cases with zero I/O
- **crawler.AC8.2 Success:** Fingerprint unit tests verify determinism, ordering invariance, and change detection
- **crawler.AC8.3 Success:** Integration tests use temp directory trees and mocked HTTP
- **crawler.AC8.4 Success:** Test fixtures follow existing project conventions (class-based grouping, factory helpers)

## Glossary

- **scan_root**: A configured base directory on the network share. The crawler walks all subdirectories from this root looking for delivery directories. Multiple scan_roots can be configured.
- **dp_id**: Data provider identifier — a 3-8 character code embedded in the directory name identifying which data provider submitted the delivery (e.g., `mkscnr`).
- **msoc / msoc_new**: Terminal directory names that signal QA status. `msoc` means QA has passed; `msoc_new` means the delivery is pending QA review.
- **delivery_id**: A deterministic SHA-256 hex digest of the `source_path`. Used as the primary key for a delivery in the registry, as the crawl manifest filename, and ensures consistency between crawler and API without coordination.
- **fingerprint**: A SHA-256 hash of a delivery's file inventory (filenames, sizes, modification times — not file contents). Used to detect whether a delivery has changed between crawl runs without reading potentially gigabytes of SAS data.
- **crawl manifest**: A JSON file written per delivery on every crawl, capturing the parsed metadata, file inventory, fingerprint, and timestamp. Serves as point-in-time audit evidence of what the crawler observed.
- **error manifest**: A JSON file written for directories that match the expected terminal names (`msoc`/`msoc_new`) but whose paths cannot be parsed into valid metadata. Provides an audit trail for parse failures.
- **SAS7BDAT**: The binary file format used by SAS statistical software. The files being inventoried (and eventually converted to Parquet) are in this format.
- **Parquet**: A columnar binary file format commonly used in data engineering. The eventual target format for the converted SAS data — not involved in the crawler itself.
- **Functional Core / Imperative Shell (FC/IS)**: An architectural pattern where pure functions with no side effects form the "core" (parsing, computation, data construction), and all I/O — filesystem, network, logging — is pushed to an outer "shell." Makes the core trivially testable.
- **RegistryUnreachableError**: A custom exception raised when all HTTP retry attempts to the registry API are exhausted. Triggers a non-zero exit, signalling to any calling process that the crawl failed.
- **exponential backoff**: A retry strategy where each successive retry waits twice as long as the previous one (here: 2s, 4s, 8s). Reduces load on a struggling service while still recovering from transient failures.
- **JSON lines**: A logging format where each log event is a single self-contained JSON object on its own line. Machine-parseable and grep-friendly, unlike multi-line structured formats.
- **idempotent**: Safe to run multiple times with the same inputs and get the same outputs and side effects. Re-crawling an unchanged delivery produces an identical manifest and no registry state change.
- **DeliveryCreate**: The Pydantic request model the registry API expects on `POST /deliveries`. The crawler constructs and posts this payload for each parsed delivery.
- **urllib.request**: Python's stdlib HTTP client. Used here instead of `requests` or `httpx` to avoid adding a runtime dependency for what is essentially a single POST call.

## Architecture

Functional Core / Imperative Shell pipeline. Pure functions handle parsing, fingerprinting, and manifest construction. A thin imperative shell handles filesystem I/O, manifest writing, and HTTP calls to the registry API.

### Module Structure

```
src/pipeline/crawler/
  __init__.py
  parser.py        # FC: parse_path(), path validation
  fingerprint.py   # FC: compute_fingerprint(file_list)
  manifest.py      # FC: build_manifest(), build_error_manifest()
  http.py          # IS: POST to registry API with retry/backoff
  main.py          # IS: walk dirs, orchestrate pipeline, entry point
```

### Data Flow

```
walk_roots(scan_roots)
  → for each msoc/msoc_new directory found:
    → parse_path(path, scan_root, exclusions)
      → if excluded dp_id: skip silently
      → if parse failure: build_error_manifest() → write to errors/
      → if success: continue
    → inventory_files(source_path)          # stat .sas7bdat files
    → compute_fingerprint(file_inventory)   # sha256 of sorted mtime+size
    → build_manifest(parsed, inventory, fingerprint, version)
    → write manifest to crawl_manifests/<delivery_id>.json
    → post_delivery(api_url, payload)
      → if RegistryUnreachableError: log, exit non-zero
```

### Component Responsibilities

**parser.py** (Functional Core): Takes an absolute path string, a scan_root string, and a set of dp_id exclusions. Returns one of three results: parsed metadata (dataclass with request_id, project, request_type, workplan_id, dp_id, version, qa_status, source_path, scan_root), `None` for excluded dp_ids, or a parse error object (raw path + reason string). Zero I/O.

Parsing strategy: given a version directory name like `soc_qar_wp001_mkscnr_v01`, match `_[a-zA-Z0-9]{3,8}_v\d+$` from the right. Everything before that match is the request_id. Then split request_id to extract project, wp_type (request_type), and workplan_id. QA status determined by terminal directory name: `msoc` = passed, `msoc_new` = pending.

**fingerprint.py** (Functional Core): Takes a list of file inventory entries (filename, size_bytes, modified_at). Sorts by filename, concatenates `filename:size_bytes:modified_at` for each, SHA-256 hashes the result. Returns `"sha256:<hex>"`. Deterministic — same input always produces same output. Does not read file contents (mtime+size is sufficient for change detection without the I/O cost of hashing gigabytes of SAS data).

**manifest.py** (Functional Core): Two functions. `build_manifest()` takes parsed metadata, file inventory, fingerprint, and crawler version — returns a dict matching the crawl manifest schema. `build_error_manifest()` takes raw path, scan_root, error reason, and crawler version — returns a dict for the error manifest schema. Neither function touches the filesystem.

Delivery ID is computed as SHA-256 hex of source_path — same deterministic algorithm the registry API uses. Manifest filename and registry delivery_id are always in sync.

**http.py** (Imperative Shell): `post_delivery(api_url, payload)` sends a DeliveryCreate JSON payload to `POST /deliveries` using `urllib.request`. Returns response body dict on success. Retries 3 times with exponential backoff (2s, 4s, 8s) on connection errors and 5xx responses. Does not retry 4xx (payload bug, retrying won't help). Raises `RegistryUnreachableError` after all retries exhausted.

**main.py** (Imperative Shell): Entry point for `python -m crawler.main`. Loads config, sets up structured JSON logger, walks each scan_root looking for `msoc`/`msoc_new` directories, orchestrates the pipeline for each candidate, handles abort on API failure, logs summary, exits with appropriate status code.

### Crawl Manifest Schema

```json
{
  "crawled_at": "2026-04-09T14:30:00Z",
  "crawler_version": "1.0.0",
  "delivery_id": "a7f3b2...",
  "source_path": "/requests/qa/mkscnr/packages/soc_qar_wp001/soc_qar_wp001_mkscnr_v01/msoc",
  "scan_root": "/requests/qa",
  "parsed": {
    "request_id": "soc_qar_wp001",
    "project": "soc",
    "request_type": "qar",
    "workplan_id": "wp001",
    "dp_id": "mkscnr",
    "version": "v01"
  },
  "qa_status": "passed",
  "fingerprint": "sha256:e4d909...",
  "files": [
    { "filename": "enrollment.sas7bdat", "size_bytes": 1048576, "modified_at": "2026-04-08T09:15:00Z" }
  ],
  "file_count": 47,
  "total_bytes": 2147483648
}
```

Written to `pipeline/crawl_manifests/<delivery_id>.json`. Overwritten on every crawl (idempotent).

### Error Manifest Schema

```json
{
  "error_at": "2026-04-09T14:30:00Z",
  "crawler_version": "1.0.0",
  "raw_path": "/requests/qa/something/packages/weird_thing/...",
  "scan_root": "/requests/qa",
  "error": "could not extract version segment from directory name"
}
```

Written to `pipeline/crawl_manifests/errors/<sha256-of-raw-path>.json`. Hash-based filename ensures idempotent overwrites for the same bad path.

### Configuration Additions

New fields in `pipeline/config.json` (and corresponding `PipelineConfig` dataclass updates):

- `dp_id_exclusions`: list of strings, initially `["nsdp"]`
- `crawl_manifest_dir`: string, defaults to `"pipeline/crawl_manifests"`
- `crawler_version`: string, e.g. `"1.0.0"`

### Structured JSON Logging

Establishes the project's first logging pattern. JSON lines format via stdlib `logging` with a custom formatter. Output to both `pipeline/logs/crawler.log` (from config `log_dir`) and stderr (for cron capture). Fields: `timestamp`, `level`, `message`, plus contextual fields (`scan_root`, `source_path`, `delivery_id`) when available.

## Existing Patterns

Investigation found the project follows Functional Core / Imperative Shell consistently. The crawler's `__init__.py` already has `# pattern: Functional Core`. This design follows that pattern — pure functions in parser.py, fingerprint.py, manifest.py; imperative shell in main.py and http.py.

Config access follows the lazy-load pattern via `pipeline.config.settings`. The crawler uses this directly — no new config loading mechanism.

The registry API's `DeliveryCreate` model accepts all string fields without format validation. All format enforcement (dp_id length, version format, request_id structure) happens in the crawler's parser before POSTing.

Test conventions from the registry API: class-based test grouping, factory helper functions (`make_delivery_payload(**overrides)` pattern), in-memory fixtures. Crawler tests follow the same conventions.

No existing logging pattern was found. This design introduces structured JSON logging as the first logging implementation. The converter and registry API can adopt the same formatter later.

## Additional Considerations

**Manifest as audit trail:** Crawl manifests exist because the registry tracks state but not evidence. In healthcare data pipelines, "what did we actually see and when" is audit trail, not nice-to-have. The converter can diff manifest file lists against what's on disk at conversion time to detect changes between crawl and conversion.

**Fingerprint scope:** Fingerprint is mtime+size only, not file content hash. This is deliberate — hashing file contents would mean reading potentially gigabytes of SAS data on every crawl cycle. mtime+size catches real changes (new files, modified files, deleted files) without the I/O cost. If content-level integrity checking is needed later, that belongs in the converter (which reads the files anyway).

**dp_id exclusion evolution:** Currently a static list in config. If exclusion logic becomes more complex (patterns, conditional exclusions), consider promoting to a separate config file. For now, a JSON array in config.json is sufficient.
